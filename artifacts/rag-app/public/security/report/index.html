Public security reporting

Report a security concern

Tell the maintainers privately so the issue can be investigated before technical details become public.

Open a private GitHub report Review repository security
Keep sensitive material out of the report. Use synthetic data. Do not submit customer content, personal data, payment-card data, credentials, session tokens, API keys, or private database output.

01

Use a private channel

GitHub private vulnerability reporting is the approved intake path. Do not open a public issue with exploit details.

If private reporting is unavailable, contact the repository owner through GitHub without disclosing the vulnerability, and ask for a private channel.

02

What to include

  • The affected route, component, or commit.
  • The security boundary you expected.
  • A minimal reproduction using synthetic data.
  • The likely impact and any temporary mitigation.

03

Test without causing harm

  • Use accounts, programs, documents, and credentials you control.
  • Do not access data that is not yours.
  • Do not degrade a shared service.
  • Do not test model or infrastructure providers without permission.

04

Response and coordinated disclosure

Maintainers will acknowledge the report, confirm whether it can be reproduced, and coordinate disclosure after affected users have a reasonable chance to update.

A fix date depends on severity, reproduction, and deployment ownership. No response-time or remediation-time SLA is represented here.

Security claim boundary

Public guidance is not an assurance report

Source code, configuration, and local tests do not prove that a deployed environment is secure or compliant. Truenote is not represented here as PCI DSS compliant, FedRAMP compliant, independently assessed, or certified.
