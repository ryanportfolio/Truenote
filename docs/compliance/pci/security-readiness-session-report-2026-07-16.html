Security-team feedback and our disposition

What we agree with

  • No reviewed evidence proves a true independent AI red team, application penetration test, or segmentation test.
  • Repository code and internal tests do not prove deployed effectiveness, live monitoring, incident handling, or production scale.
  • PCI is the primary integration concern if TrueNote enters or can affect the existing CDE.
  • Requirement 6 evidence needs an operating SDLC, vulnerability process, and change-control receipts, not policy files alone.
  • Public words such as “verified” need an explicit evidence scope so buyers do not read them as external assurance.

What needs qualification or correction

  • “No real-time PII redaction firewall” is no longer accurate for the repository. OpenRouter guardrail configuration exists, local synchronous checks protect OpenAI embeddings, Cohere query/document reranking, and OpenRouter generation/rewrite/naming text, and generated answers fail closed on deterministic credential/private-key, SSN, and valid payment-card findings. LandingAI/raw files, names, addresses, contextual or obfuscated PII, broader output policy, deployment, and independent bypass testing remain open.
  • A dedicated branded “prompt firewall” is not itself a PCI requirement. The control objectives, coverage, safe failure, monitoring, bypass testing, and evidence matter.
  • PCI is not a universal standalone product certification or universal statute. The likely path is inclusion in the customer’s existing assessed CDE, with scope and validation decided by the compliance-accepting entity and QSA.
  • “Never tested” or “never audited” is stronger than public evidence supports. The defensible statement is “no independent report was reviewed.”
  • Static repository inspection can support “implemented in reviewed source,” but not “correctly built” across runtime, configuration, and adversarial conditions.

General lessons

  • A safe inventory is not vulnerability management until people own dates, decisions, exceptions, and retests.
  • A finding disappearing from a scanner is evidence to investigate, not permission to close it.
  • Stable IDs and retained history matter as much as current counts because auditors sample decisions over time.
  • Evidence metadata needs provenance and concurrency protection. A structurally valid JSON file can still be misleading if it came from the wrong run or overwrote a newer decision.
  • Internal tests can verify implementation behavior. They cannot become “independent” by changing their label.
  • Security claims should state the evidence boundary in the same sentence: repository test, configuration screenshot, deployed receipt, provider attestation, or third-party report.
  • “Required code-owner review” is not meaningful evidence until CODEOWNERS actually covers every applicable path.
  • A historical settings result must not be presented as current after live authentication fails. Record both events and keep current state unverified.
  • Branch-setting metadata can be machine-checked, but effective enforcement needs complete authenticated API bodies, ruleset reconciliation, and denied merge/push tests.
  • A link and sitemap entry do not prove a public route exists. Retain the physical source, portable content/discovery tests, deployed response evidence, and an operating intake sample.
